- Data ingestion from raw CSV files
- Validation checks for schema and data quality

## 8. Technical Findings from Full Folder Analysis

### Strengths

1. Dual processing capability
- The repository supports both pandas-centric and PySpark-centric workflows.

2. Data quality mindset
- Validation checks exist in both notebooks before deeper analysis.

3. Output discipline
- Generated artifacts are organized under outputs/ with clear naming.

4. Documentation baseline
- README and folder-level READMEs provide setup orientation.

### Risks and Gaps

1. Hardcoded secrets in code (High)
- PostgreSQL credentials appear in notebook/script source.

2. Hybrid notebook coupling (Medium)
- 01_eda_exploration.ipynb mixes CSV load and DB query logic in the same flow, making execution assumptions implicit.

3. Missing dependency declarations (High)
- SQLAlchemy/driver not pinned in requirements.txt.

4. Reproducibility friction (Medium)
- Notebook execution depends on local PostgreSQL state if SQL cells are run.

5. No automated tests (Medium)
- Validation exists at runtime but there is no test suite for regression checks.

6. README duplication/noise (Low)
- README currently has repeated sections and can be tightened for maintainability.

## 9. Recommended Change Plan (Prioritized)

### Priority 1 - Security and Reproducibility

1. Remove hardcoded DB credentials
- Use environment variable, for example RETAIL_DB_URL.
- Provide .env.example (without secrets).

2. Update requirements.txt
- Add sqlalchemy
- Add psycopg2-binary (or chosen Postgres driver)

3. Add explicit run modes in docs
- CSV-only mode
- PostgreSQL-enabled mode

### Priority 2 - Pipeline Clarity

1. Separate concerns in notebook/script workflow
- Keep one pure pandas CSV path
- Keep one DB-backed analysis path

2. Add preflight checks
- Validate DB connectivity before SQL queries
- Validate local data paths before load

### Priority 3 - Engineering Quality

1. Introduce minimal src/ module structure
- ingestion.py
- validation.py
- transform.py
- reporting.py

2. Add smoke tests
- Dataset load test
- Join-row integrity test
- Validation report generation test

## 11. How to Run (As-Is)

### Option A: CSV + Notebook flow
1. Place Walmart CSV files using the documented data/ structure.
2. Create venv and install dependencies.
3. Run notebook cells in order.

### Option B: PostgreSQL-backed flow
1. Create retail_db in local PostgreSQL.
2. Configure connection details.
3. Run db_setup.py to load tables.
4. Run SQL-powered cells in 01_eda_exploration.ipynb.

Note:
- Option B currently requires manual credential editing in source unless refactored.
